Telco ODS - Autonomous Network Health Pipeline
Back to Dashboard

What This Demo Shows

This is a live, end-to-end streaming ML pipeline that demonstrates MongoDB Atlas as an Operational Data Store (ODS) for autonomous network health monitoring. The pipeline ingests raw telemetry from 50 simulated cell towers, applies real-time stream processing, runs ML inference via event-driven triggers, and serves pre-computed features — all with MongoDB Atlas at the centre.

The goal is to show how Atlas handles the requirements of a modern ODS: high-throughput ingestion, flexible schema, event-driven workflows, ML integration, and low-latency operational reads — without needing a separate streaming database, feature store backend, or orchestration layer.

Pipeline Architecture

Data Generator (50 cell towers, ~17,000 events/sec) | | raw telemetry events v Apache Kafka (single broker, 12 partitions) | | consumer group v Apache Flink (PyFlink — 5-min sliding window, 30s slide, per cell) | | aggregated metrics (avg, p95, min, max per cell per window) v MongoDB Atlas — windowed_network_metrics (ODS collection) | | Atlas Database Trigger (fires on every insert, zero orchestration) v MLflow Model Serving (RandomForest classifier → excellent / good / poor) | | prediction written back v MongoDB Atlas — network_health_predictions (results collection) --- Feature Store path (parallel) --- MongoDB Atlas — windowed_network_metrics | | Feast Materialization (every 2 min, reads latest per cell) v MongoDB Atlas — telco_ods_online (Feast online store collection) | | sub-15ms point lookups v Feast Feature Server (HTTP API for downstream consumers)

Data Flow Step by Step

1
Telemetry Generation
A Python generator simulates 50 cell towers across 5 Australian regions (Sydney, Melbourne, Brisbane, Perth, Adelaide). Each tower emits network metrics — signal strength, throughput, latency, packet loss, jitter, call drop rate — at ~340 events/sec/tower. Some towers are configured to periodically degrade, simulating real-world network issues.
2
Stream Ingestion (Kafka)
Events flow into a 12-partition Kafka topic. This decouples the generator from processing, handles backpressure, and allows multiple consumers. In production this could be replaced with any event bus — the key point is raw events never touch Atlas directly.
3
Stream Processing (Flink)
PyFlink applies a 5-minute sliding window with 30-second slides, keyed by cell_id. For each window it computes rolling aggregates: mean, min, max, p95 for all metrics, plus event count and anomaly detection. The output is a single document per cell per slide written to Atlas — this is the "operational aggregate" that forms the ODS.
4
ODS Ingestion (Atlas)
Flink writes windowed metrics to windowed_network_metrics via pymongo. Each document has a flexible nested schema with per-metric statistics. Atlas handles ~100 inserts/sec (50 cells x 1 insert per 30s slide per cell, staggered).
5
Event-Driven ML Inference (Atlas Trigger)
An Atlas Database Trigger watches inserts on windowed_network_metrics. On each insert, it calls the MLflow model serving endpoint with the aggregated features. The model returns a classification (excellent / good / poor) and confidence score. The trigger writes the result to network_health_predictions. No external scheduler, no polling — purely event-driven with sub-second latency.
6
Feature Store (Feast + MongoDB Online Store)
Every 2 minutes, Feast materializes the latest per-cell features from windowed_network_metrics into a dedicated MongoDB collection (telco_ods_online). The Feast Feature Server provides a standard HTTP API for low-latency feature retrieval — demonstrating how Atlas can back both the ODS and the feature store without a separate Redis/DynamoDB layer.

ODS Requirements Demonstrated

FR-3
Real-time Ingestion
17,000+ raw events/sec processed by Flink, aggregated metrics written to Atlas every 30 seconds per cell tower with consistent sub-100ms write latency.
FR-5
Schema Flexibility
Windowed metrics use nested documents with per-metric statistics (avg, min, max, p95). No migrations needed — schema evolves with the pipeline.
FR-9
API-based Access
Dashboard reads via MongoDB Query API. Feast serves features over HTTP. Atlas Triggers provide serverless compute. No proprietary protocols.
FR-11
ML Feature Enablement
Feast Feature Store with MongoDB as online store backend. Sub-15ms feature retrieval. Same database serves both operational queries and ML features.
FR-12
Event-driven Workflows
Atlas Database Trigger fires on every insert — calls ML model, writes prediction, no orchestration layer. Change streams enable real-time reactions.
FR-14
Operational Scope
Raw data stays in Kafka (ephemeral). ODS holds the operational window: aggregated metrics, predictions, and features — right-sized for operational queries.

Technology Stack

MongoDB Atlas (ODS) Atlas Database Triggers Change Streams Feast Online Store Apache Kafka Apache Flink (PyFlink 1.18) MLflow scikit-learn (RandomForest) AWS EC2 (5 instances) Python Flask Dashboard

Infrastructure

5 EC2 instances deployed via CloudFormation in ap-southeast-2:

  • Kafka — Single broker, 12 partitions, handles all raw event throughput
  • Generator — Python process simulating 50 cell towers
  • Flink — PyFlink cluster running the windowed aggregation job
  • MLflow — Model registry, serving endpoint, and this dashboard
  • Feast — Feature server with MongoDB online store backend

Key Design Decisions

  • Atlas as single operational backend — ODS, feature store, and prediction results all in one cluster. No Redis, no DynamoDB, no separate read replicas.
  • Triggers instead of polling — ML inference is purely event-driven. No cron jobs, no Airflow DAGs, no batch scoring.
  • Sliding windows, not tumbling — 5-min window with 30s slides means predictions update every 30 seconds with 5 minutes of context. Smooth, not bursty.
  • Feast on MongoDB — Demonstrates that Atlas can serve as a feature store backend with competitive latency, eliminating a separate caching layer.
  • Operational scope — Raw events stay in Kafka (short retention). Only aggregated, operationally-relevant data enters Atlas. Right data, right layer.
