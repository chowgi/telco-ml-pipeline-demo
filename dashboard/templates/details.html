The Problem

  • Telco networks generate millions of telemetry events per second — signal strength, throughput, latency, jitter, packet loss across thousands of cell towers
  • Current approach: batch analytics with 30-60 min delay. By the time you detect a degraded tower, customers have already dropped calls
  • What's needed: real-time anomaly detection that reacts to data as it arrives — not on a schedule, not after a batch job completes
  • The ODS challenge: you need a single platform that can ingest at speed, trigger ML workflows, serve features, and answer operational queries — without stitching together 5 different systems

Pipeline Architecture

Data Generator (50 cells, ~17k events/sec) | v Apache Kafka (12 partitions) | v Apache Flink (5-min window, 30s slide, per cell) | v MongoDB Atlas — windowed_network_metrics | | Atlas Trigger (fires on insert) v MLflow Model → prediction → Atlas

Why MongoDB — Not Something Else?

  • Reactive, not passive — Atlas Triggers (Change Streams) fire ML inference the instant data lands. No Airflow, no Step Functions, no Lambda glue. Try that with Cassandra or TimescaleDB.
  • One cluster, three roles — ODS writes, feature store, and prediction store in one platform. Competitors need Cassandra + Redis + Postgres — three systems to operate.
  • Document model = zero migrations — nested per-metric stats (avg, min, max, p95) in one document. New KPI? Add a field. No ALTER TABLE, no downtime.
  • Operational + ML on the same data — Feast online store backed by Atlas. No separate caching tier for ML features. Sub-15ms retrieval from the same cluster.

Data Flow

  • Generate: 50 cell towers emit signal strength, throughput, latency, packet loss, jitter, call drops
  • Stream: Kafka decouples ingestion from processing — raw events never touch Atlas
  • Aggregate: Flink computes rolling stats (avg, min, max, p95) per cell per 30s slide
  • Store: One document per cell per window written to Atlas as the ODS
  • React: Atlas Trigger fires, calls ML model, writes prediction — zero orchestration code

Without MongoDB — What This Looks Like

  • TimescaleDB / InfluxDB — ingest works, but no native triggers. You need Lambda + EventBridge + custom glue to fire ML inference
  • Cassandra — fast writes, but no aggregation, no change streams, no event-driven compute. You bolt on Spark + Airflow
  • Feature store — Redis or DynamoDB as a separate system for ML serving. More infra, more latency, more ops
  • MongoDB Atlas — ingest, react, serve, query. One platform. The ODS does things, it doesn't just store things

Key Design Decisions

  • Atlas as single backend — ODS, feature store, predictions in one cluster
  • Triggers, not polling — ML inference is purely event-driven, sub-second response
  • Sliding windows — 5-min context, 30s updates. Smooth signal, not bursty snapshots
  • Operational scope — raw data stays in Kafka, only aggregates in Atlas. Right data, right layer
